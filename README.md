# MoDeRo — предиктивная аналитика состояния крутильного вала турбоагрегата

Учебно-/научно-исследовательская работа (УИР/НИР). Диагностика состояния
**крутильного вала** турбоагрегата по 1-й собственной частоте крутильных
колебаний `F1` (Гц) в переходных режимах работы нагрузки.

Подход — **Вариант C (гибрид)**: F1(t) = `f_reg(x(t))` + `r(t)`, где
`f_reg` — режимная модель F1 по нагрузке/температурам, а остаток `r(t)` —
индикатор «здоровья» вала (устойчивый дрейф = ранний признак дефекта).
Полная постановка, данные и методология — в `project_brief_MoDeRo.md`
(утверждённый источник правды).

## Данные

Положите два исходных файла в `data/` (etl ищет их по шаблону имени):

- **Файл A** (сырой, основной): `Данные ПТК МоДеРо за 2015-2016гг.xlsx`
  — ~12 470 почасовых строк, 04.03.2015 → 18.01.2017.
- **Файл B** (Prism, для бенчмарка): `Prism_МоДеРо.xlsx`
  — 6 679 строк (подпериод A), остатки эталонной модели Prism.

## Отклонение от плана: порядок этапов 7 → 6 (обоснование)

Бриф предполагает порядок 5 → 6 (прогноз) → 7 (валидация детектора). В работе
этапы **поменяны местами: сначала 7 (синтетический дрейф), затем 6 (прогноз)**.

**Обоснование.** Единственный наблюдаемый эпизод дрейфа (авг–сен 2016) частично
конфаундится сезоном: лето 2015 (in-train, OOS) тоже даёт положительный сдвиг
остатка (+0.0033 Гц), хотя эпизод 2016 в 4.4× больше (+0.0144 Гц, разница
статистически несомненна, t≈28). На наблюдательных данных детектор даёт по сути
**одну** точку out-of-sample-детекции (2016-08-31) и не позволяет измерить его
характеристики (задержку обнаружения, минимальный обнаружимый дрейф, ROC).
Пока детектор не охарактеризован контролируемым впрыском (этап 7), прогноз
индикатора H(t+h) (этап 6) опирается на неподтверждённый сигнал. Поэтому сначала
валидируем детектор синтетикой, затем строим прогноз.

## Чистая таблица `outputs/modero_clean.csv`

Одна строка = один час. Ключевые колонки:

| колонка                       | смысл                                                   |
|-------------------------------|---------------------------------------------------------|
| `time`                        | почасовая метка времени (индекс)                        |
| `F1`                          | целевая: 1-я СЧ ККВ, Гц                                 |
| `Pa, Pr`                      | активная/реактивная мощность                            |
| `Td, Tg, Tst`                 | температуры дистиллята/пара/статора                     |
| `Tcnd1, Tcnd2`                | Тп.ЦНД-1/2                                               |
| `Tgpz_a..g`                   | Т ГПЗ н.А–Г                                              |
| `Tturb_a..g`                  | Т турб. н.А–Г                                            |
| `Fr`                          | частота вращения, Гц                                    |
| `seg`                         | id непрерывного сегмента (разрыв шага ≠ 1 ч → новый seg) |
| `dPa`                         | ΔPa за час, **внутри сегмента** (первый час сегмента — NaN) |
| `transient`                   | флаг переходного часа: `\|ΔPa\| > 20 МВт`                |
| `resid_prism`                 | остаток модели Prism на перекрытии (иначе NaN)          |

> **Лаги, скользящие и рампы строятся ТОЛЬКО внутри сегментов** (`groupby('seg')`) —
> иначе утечка через разрывы. Временное разбиение строго по разделу 5 брифа,
> без перемешивания.

## Замечания по данным (выявлено на ETL) — материал для практического раздела ПЗ

Этап ETL вскрыл два скрытых дефекта в исходном Prism-файле. Это хороший
«бонусный» материал в практический раздел ПЗ (тема: *data hygiene* — ловля
скрытых дефектов в исходных данных до моделирования).

1. **Псевдо-нули в остатке Prism = «не посчитано», а не «остаток 0».**
   `resid_prism` заполнен лишь в **1219** из 6679 часов перекрытия. Остальные
   5460 значений — ровно `0`, разбросаны по всему диапазону (не сплошным блоком).
   Для остатка модели с 6-значной F1 значение `0.00000000` физически невозможно
   — это маркер отсутствия прогноза Prism на данный час. Решение: `0 → NaN`.
   **Последствие для метрик:** усреднение с нулями занижает RMSE остатка
   (ложные `0.0021 Гц` на всех 6679 ч против честных **`0.0050 Гц`** на 1219
   реально посчитанных часах). Reference-цифра проекта: **RMSE_Prism = 0.0050 Гц
   на N = 1219 ч** (в роли контекста, не цели, которую нужно обойти).

2. **«ДП1» — не независимая модель, а масштабированная копия Prism.**
   Колонка с заголовком «ДП1» = остаток Prism × постоянный коэффициент
   `≈ 517.72` (std отношения = 0, те же 1219 ненулевых позиций; вероятно, другая
   единица измерения / форма для отображения). Независимого ряда остатков ДП1 в
   данных нет, поэтому **ДП1 не используется как отдельный baseline**. В `eval.py`
   остаются только два эталона: **persistence** (на всех часах) и **Prism**
   (на подмножестве N = 1219 ч, где модель посчитана).
