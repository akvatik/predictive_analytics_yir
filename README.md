# MoDeRo — предиктивный мониторинг ранней деградации крутильного вала турбоагрегата

Учебно-/научно-исследовательская работа (УИР/НИР). **Исследовательский
мониторинг ранней деградации** валопровода по 1-й собственной частоте крутильных
колебаний `F1` (Гц) — обнаружение устойчивого отклонения F1 от ожидаемого
режимного поведения. Это **не** диагностика конкретных поломок и **не** аварийная
защита (см. «Постановка задачи» и «Ограничения» ниже).

Подход — **Вариант C (гибрид)**: F1(t) = `f_reg(x(t))` + `r(t)`, где
`f_reg` — режимная модель F1 по нагрузке/температурам, а остаток `r(t)` —
индикатор «здоровья» вала (устойчивый дрейф остатка потенциально совместим со
снижением крутильной жёсткости — ранний сигнал изменения состояния).
Полная постановка, данные и методология — в `project_brief_MoDeRo.md`
(утверждённый источник правды).

## Постановка задачи

Мы исследуем состояние валопровода турбоагрегата по одному параметру — F1, первой
собственной частоте крутильных колебаний вала. Работающий вал способен слегка
«подкручиваться» туда-обратно, и у этих колебаний есть главная частота — это F1.
Она зависит от жёсткости вала: жёстче вал — выше частота.

В нормальной работе F1 тоже немного меняется из-за нагрузки и температур. Поэтому:
(1) строим модель, которая по текущему режиму говорит, какой F1 **ожидается**;
(2) вычитаем фактический F1 минус ожидаемый — получаем **остаток**; (3) следим за
остатком: если он устойчиво уходит в минус и держится там, это может означать, что
вал стал чуть менее жёстким — ранний признак изменения состояния.

**Проект делает:** обнаруживает устойчивое отклонение F1 от ожидаемого режимного
поведения и сигналит «параметр ведёт себя необычно, стоит присмотреться /
запланировать осмотр».

**Проект не делает:** не диагностирует конкретную поломку (по одному F1 это
доказать нельзя) и не является аварийной защитой (не ловит быстрые опасные события
за секунды/минуты).

Коротко: это инструмент раннего предупреждения о медленном изменении состояния, а
не диагностика поломок и не аварийная сигнализация.

## Физическая интерпретация F1

Вал — как тяжёлая пружина на скручивание. Её собственная частота зависит от
жёсткости (насколько трудно закрутить) и инерции (насколько массивный): жёстче —
выше частота. При развитии поперечной трещины, усталостной деградации или
ослаблении муфты вал становится чуть менее жёстким, и по физике F1 должна немного
снижаться. Поэтому устойчивое снижение F1, не объяснимое режимом, в принципе
совместимо со снижением крутильной жёсткости.

Два важных «но»: (1) изменение маленькое — ранняя слабая деградация даёт крошечный
сдвиг F1, заметный провал был бы только при поздней сильной потере жёсткости; F1 по
природе — индикатор малых и медленных изменений и не может дать большой быстрый
сигнал на ранней стадии. (2) F1 меняется не только от состояния вала — на неё влияют
нагрузка, температуры, частота сети, поэтому «ожидаемый» F1 моделируется по режиму.

Главное: один F1 говорит, **что** что-то изменилось, но не **что именно** —
однозначно указать на конкретный дефект по одному параметру нельзя.

## Ограничения

1. **Нет реальных размеченных отказов.** Детектор проверялся на искусственном
   (синтетическом) дрейфе и одном спорном реальном эпизоде. Показано, что метод
   срабатывает на известных тестовых сигналах, — но не доказано, что он ловит
   реальную деградацию вала.
2. **«Ожидаемый» F1 моделируется неточно.** Связь режим→F1 со временем меняется,
   поэтому остаток шумный — разброс около σ ≈ 0.012 Гц (σ — уровень шума).
3. **Задержка обнаружения большая, и это не баг.** Искомые сдвиги (~0.005–0.015 Гц)
   сопоставимы с шумом (~0.012 Гц). Когда сигнал не больше шума, его нельзя надёжно
   поймать быстро — детектор копит данные неделями. Это предел отношения
   сигнал/шум, а не недоработка кода. Ускорить честно можно лишь: чище остаток,
   более сильное изменение, либо больше ложных тревог.
4. **Шум и сигнал переплетены.** Часть «шума» может сама быть медленным дрейфом,
   поэтому «здоровое» окно для настройки порогов, возможно, уже содержит немного
   дрейфа — граница «шум vs ранняя деградация» размыта.
5. **Сезонный след.** Единственный реальный эпизод (лето 2016) совпал с жарой и
   горячей охлаждающей водой; полностью отделить «здоровье» от сезона в одном
   эпизоде нельзя.
6. **Один параметр — не диагноз.** Для промышленной диагностики нужны и другие
   признаки (амплитуды собственных частот, демпфирование, статическая закрутка,
   НДС, спектры, экспертная разметка). У нас только F1.

**Вывод.** Проект — исследовательский health-индикатор для раннего предупреждения о
медленных изменениях, а не промышленная аварийная защита. Срабатывание означает
«стоит присмотреться и запланировать контроль», а не «вал сломан».

## Отклонение от плана: порядок этапов 7 → 6 (обоснование)

Бриф предполагает порядок 5 → 6 (прогноз) → 7 (валидация детектора). В работе
этапы **поменяны местами: сначала 7 (синтетический дрейф), затем 6 (прогноз)**.

**Обоснование.** Единственный наблюдаемый эпизод дрейфа (авг–сен 2016) частично
конфаундится сезоном: лето 2015 (in-train, OOS) тоже даёт положительный сдвиг
остатка (+0.0033 Гц), хотя эпизод 2016 в 4.4× больше (+0.0144 Гц, разница
статистически несомненна, t≈28). На наблюдательных данных детектор даёт по сути
**одну** точку out-of-sample-детекции (2016-08-31) и не позволяет измерить его
характеристики (задержку обнаружения, минимальный обнаружимый дрейф, ROC).
Пока детектор не охарактеризован контролируемым впрыском (этап 7), прогноз
индикатора H(t+h) (этап 6) опирается на неподтверждённый сигнал. Поэтому сначала
валидируем детектор синтетикой, затем строим прогноз.

```bash
pip install -r requirements.txt

# Этап 1 — ETL: собрать чистую почасовую таблицу
python -m src.etl
# → outputs/modero_clean.csv  (+ сводка сегментов / переходных часов / перекрытия)

# Этап 3 — признаки (после ETL)
python -m src.features
# → outputs/modero_features.parquet

# Слой 1 (этап 4) → слой 2 (этап 5) → синтетика (этап 7) → прогноз (этап 6)
python -m src.models.gbm        # → layer1_metrics.csv, modero_residuals.parquet
python -m src.health            # → health_series.parquet, health_indicator.png
python -m src.synthetic_drift   # → synthetic_drift_delays.csv, synthetic_roc.png
python -m src.forecast          # → layer3_forecast_metrics.csv
python -m src.transient_study   # этап 8 → transient_study.csv/.png

# Предиктивный слой (переосмысление постановки)
python -m src.risk              # TTT + уровни риска → time_to_threshold.csv, risk_timeline.csv
python -m src.sensitivity_noise # what-if по σ → sensitivity_noise.csv/.png
python -m src.scenarios_table   # таблица сценариев → synthetic_scenarios_table.csv
```

> **Предиктивные артефакты** (раннее предупреждение, не диагностика):
> `time_to_threshold.csv`/`.png` — оценка «через сколько дней дойдёт до порога» при
> значимом отрицательном тренде; `risk_timeline.csv` — уровни Green/Yellow/Orange/Red
> (Red лишь документирует границу вне области применимости); `sensitivity_noise.*` —
> what-if «задержка vs σ»; `synthetic_scenarios_table.csv` — физическая интерпретация
> синтетических сценариев. Все пороги — из уже откалиброванного `health_series.parquet`.

## Чистая таблица `outputs/modero_clean.csv`

Одна строка = один час. Ключевые колонки:

| колонка                       | смысл                                                   |
|-------------------------------|---------------------------------------------------------|
| `time`                        | почасовая метка времени (индекс)                        |
| `F1`                          | целевая: 1-я СЧ ККВ, Гц                                 |
| `Pa, Pr`                      | активная/реактивная мощность                            |
| `Td, Tg, Tst`                 | температуры дистиллята/пара/статора                     |
| `Tcnd1, Tcnd2`                | Тп.ЦНД-1/2                                               |
| `Tgpz_a..g`                   | Т ГПЗ н.А–Г                                              |
| `Tturb_a..g`                  | Т турб. н.А–Г                                            |
| `Fr`                          | частота вращения, Гц                                    |
| `seg`                         | id непрерывного сегмента (разрыв шага ≠ 1 ч → новый seg) |
| `dPa`                         | ΔPa за час, **внутри сегмента** (первый час сегмента — NaN) |
| `transient`                   | флаг переходного часа: `\|ΔPa\| > 20 МВт`                |
| `resid_prism`                 | остаток модели Prism на перекрытии (иначе NaN)          |

> **Лаги, скользящие и рампы строятся ТОЛЬКО внутри сегментов** (`groupby('seg')`) —
> иначе утечка через разрывы. Временное разбиение строго по разделу 5 брифа,
> без перемешивания.

## Замечания по данным (выявлено на ETL) — материал для практического раздела ПЗ

Этап ETL вскрыл два скрытых дефекта в исходном Prism-файле. Это хороший
«бонусный» материал в практический раздел ПЗ (тема: *data hygiene* — ловля
скрытых дефектов в исходных данных до моделирования).

1. **Псевдо-нули в остатке Prism = «не посчитано», а не «остаток 0».**
   `resid_prism` заполнен лишь в **1219** из 6679 часов перекрытия. Остальные
   5460 значений — ровно `0`, разбросаны по всему диапазону (не сплошным блоком).
   Для остатка модели с 6-значной F1 значение `0.00000000` физически невозможно
   — это маркер отсутствия прогноза Prism на данный час. Решение: `0 → NaN`.
   **Последствие для метрик:** усреднение с нулями занижает RMSE остатка
   (ложные `0.0021 Гц` на всех 6679 ч против честных **`0.0050 Гц`** на 1219
   реально посчитанных часах). Reference-цифра проекта: **RMSE_Prism = 0.0050 Гц
   на N = 1219 ч** (в роли контекста, не цели, которую нужно обойти).

2. **«ДП1» — не независимая модель, а масштабированная копия Prism.**
   Колонка с заголовком «ДП1» = остаток Prism × постоянный коэффициент
   `≈ 517.72` (std отношения = 0, те же 1219 ненулевых позиций; вероятно, другая
   единица измерения / форма для отображения). Независимого ряда остатков ДП1 в
   данных нет, поэтому **ДП1 не используется как отдельный baseline**. В `eval.py`
   остаются только два эталона: **persistence** (на всех часах) и **Prism**
   (на подмножестве N = 1219 ч, где модель посчитана).
